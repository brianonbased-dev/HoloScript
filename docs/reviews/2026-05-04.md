## Findings

### P2 — `32d7608` — Simulated tracking metrics displayed as real data

**File**: `packages/studio/src/app/scan-room/mobile/[token]/page.tsx`

The HoloMap camera overlay renders a "Tracking mesh" progress bar and point count that are entirely time-based approximations, not real tracking data:

```ts
const trackingPercent =
  cameraState === 'recording'
    ? Math.min(100, 24 + recordingSeconds * 8)   // ← fabricated formula
    : cameraState === 'ready' ? 18 : ...

const overlayPointCount = Math.max(
  sessionFeedback?.renderAsset?.pointCount ?? 0,
  cameraState === 'recording' ? 128 + recordingSeconds * 36 : 0,  // ← fabricated
);
```

The overlay also labels three columns as "Floor plane", "Wall edges", and a point count — all hardcoded UI chrome with no real signal behind them. A user filming a blank wall will see 100% tracking and thousands of points.

**Risk**: Misleads users into thinking the scan is actively working when it may not be. If Studio later wires real tracking signals, these fabricated fallbacks will silently mask them if `recordingSeconds * 36 > realPointCount`.

**Recommendation**: Either label the progress bar clearly as "Recording…" (time-based timer), or remove the simulated point count injection from the `Math.max`. The real `sessionFeedback?.renderAsset?.pointCount` already shows 0 pts until a render asset exists, which is honest.

### P2 — `c36364c` — Two capture file inputs rendered simultaneously when live camera is available

**File**: `packages/studio/src/app/scan-room/mobile/[token]/page.tsx`

When `cameraCapability.live === true` and `cameraState === 'idle'`, both of these are shown:
1. A primary `<label>` with a hidden `<input type="file" capture="environment" data-testid="room-camera-input">` (inside the idle section)
2. A secondary fallback `<label>` with `<input type="file" capture="environment" data-testid="room-camera-input-fallback">` (outside the idle section, conditionally on `cameraCapability.live`)

This means two file inputs with `capture="environment"` co-exist in the DOM for the HTTPS/live path. On iOS Safari, triggering either one opens the camera. The labels have different text ("Open phone camera" vs "Use phone camera app fallback") but they do identical things. This is dead UI surface that could confuse users and complicates Playwright targeting.

**Recommendation**: On the live path, show only one file input for fallback; the primary CTA should be the "Use live preview" button. Remove the `capture` input from the primary label when `cameraCapability.live` is true.

### P2 — `d624371` — No test coverage for new `storage` fields in `/health` response

**File**: `packages/mcp-server/src/http-server.ts`

The `/health` endpoint now returns:
```json
{ "storage": { "teamStore": "postgres" | "memory", "databaseUrl": true | false } }
```

The `teamStore.usesPostgres` getter was wired specifically to diagnose Pattern Gamma (state evaporation on redeploy). However, there is no test asserting these fields appear in the health response, nor that the `teamStore` value correctly reflects backend selection. A regression that re-introduces the in-memory fallback would go undetected.

**Recommendation**: Add a unit test in `packages/mcp-server/src/__tests__/` (or the existing http-routes test file) that mocks `teamStore.usesPostgres` true/false and asserts the health response includes the correct `storage.teamStore` string.

### P3 — `99efd4b` — Fixed-interval polling with no error backoff

**File**: `packages/studio/src/app/scan-room/mobile/[token]/page.tsx`

The session feedback `useEffect` polls `/api/reconstruction/session` every 1200 ms with no backoff on failure:

```ts
const interval = window.setInterval(() => void refreshFeedback(), 1200);
```

On error, `setFeedbackError` is called but the next poll fires 1200 ms later unconditionally. If the Studio server is unreachable (e.g., user switches networks mid-scan), the phone will generate ~3000 failed requests per hour until the page is closed.

**Recommendation**: Implement simple exponential backoff on consecutive errors (e.g., cap at 10s after 3 consecutive failures), or stop polling once `done === true` or `error !== null`.

### P3 — `9ed9c67` — `captureNotice` not cleared on error path; potential dual-message display

**File**: `packages/studio/src/app/scan-room/mobile/[token]/page.tsx`

When an upload fails, `captureNotice` retains its last value (e.g., `'Preparing video metadata...'`). The error path sets `setError(message)` but never calls `setCaptureNotice(null)`. After the `finally` block sets `uploading` to `false`, the condition `captureNotice && !uploading && !done` is true, so the stale notice is shown alongside the error banner. This creates two simultaneous status messages with contradictory signals.

**Recommendation**: Add `setCaptureNotice(null)` to the `catch` block in `submitCapture`.
