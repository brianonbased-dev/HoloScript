### P2 вЂ” `32d7608` вЂ” Simulated tracking metrics displayed as real data

**File**: `packages/studio/src/app/scan-room/mobile/[token]/page.tsx`

The HoloMap camera overlay renders a "Tracking mesh" progress bar and point count that are entirely time-based approximations, not real tracking data:

```ts
const trackingPercent =
  cameraState === 'recording'
    ? Math.min(100, 24 + recordingSeconds * 8)   // в†ђ fabricated formula
    : cameraState === 'ready' ? 18 : ...

const overlayPointCount = Math.max(
  sessionFeedback?.renderAsset?.pointCount ?? 0,
  cameraState === 'recording' ? 128 + recordingSeconds * 36 : 0,  // в†ђ fabricated
);
```

The overlay also labels three columns as "Floor plane", "Wall edges", and a point count вЂ” all hardcoded UI chrome with no real signal behind them. A user filming a blank wall will see 100% tracking and thousands of points.

**Risk**: Misleads users into thinking the scan is actively working when it may not be. If Studio later wires real tracking signals, these fabricated fallbacks will silently mask them if `recordingSeconds * 36 > realPointCount`.

**Recommendation**: Either label the progress bar clearly as "RecordingвЂ¦" (time-based timer), or remove the simulated point count injection from the `Math.max`. The real `sessionFeedback?.renderAsset?.pointCount` already shows 0 pts until a render asset exists, which is honest.

### P2 вЂ” `c36364c` вЂ” Two capture file inputs rendered simultaneously when live camera is available

**File**: `packages/studio/src/app/scan-room/mobile/[token]/page.tsx`

When `cameraCapability.live === true` and `cameraState === 'idle'`, both of these are shown:
1. A primary `<label>` with a hidden `<input type="file" capture="environment" data-testid="room-camera-input">` (inside the idle section)
2. A secondary fallback `<label>` with `<input type="file" capture="environment" data-testid="room-camera-input-fallback">` (outside the idle section, conditionally on `cameraCapability.live`)

This means two file inputs with `capture="environment"` co-exist in the DOM for the HTTPS/live path. On iOS Safari, triggering either one opens the camera. The labels have different text ("Open phone camera" vs "Use phone camera app fallback") but they do identical things. This is dead UI surface that could confuse users and complicates Playwright targeting.

**Recommendation**: On the live path, show only one file input for fallback; the primary CTA should be the "Use live preview" button. Remove the `capture` input from the primary label when `cameraCapability.live` is true.

### P3 вЂ” `99efd4b` вЂ” Fixed-interval polling with no error backoff

**File**: `packages/studio/src/app/scan-room/mobile/[token]/page.tsx`

The session feedback `useEffect` polls `/api/reconstruction/session` every 1200 ms with no backoff on failure:

```ts
const interval = window.setInterval(() => void refreshFeedback(), 1200);
```

On error, `setFeedbackError` is called but the next poll fires 1200 ms later unconditionally. If the Studio server is unreachable (e.g., user switches networks mid-scan), the phone will generate ~3000 failed requests per hour until the page is closed.

**Recommendation**: Implement simple exponential backoff on consecutive errors (e.g., cap at 10s after 3 consecutive failures), or stop polling once `done === true` or `error !== null`.

#### P2 вЂ” `29e9a8d` вЂ” persistTeamStore write-through and stale-read fix have no regression tests

**Files**: `packages/mcp-server/src/holomesh/state.ts`, `packages/mcp-server/src/holomesh/routes/board-routes.ts`

This commit fixes the Pattern Gamma board-wipe symptom via two changes:

1. `GET /board` now calls `requireTeamAccessFresh` instead of sync `requireTeamAccess` so Railway replicas always read from Postgres.
2. `persistTeamStore()` no longer early-returns on the Postgres path вЂ” it now iterates `teamStore.values()` and calls `teamStore.set()` to fire write-through.

Both fixes are correct. However, the existing test suite mocks `persistTeamStore` as `vi.fn()` in `board-tools.test.ts`, meaning it only verifies the function is *called*, not what it *does*. The new write-through loop in the Postgres branch is never exercised by tests. Similarly, there is no test that mocks `teamStore.getFresh` and verifies `GET /board` returns the freshly-loaded value rather than the stale cache.

The commit message itself calls out four other read endpoints (`GET /board/done`, `GET /suggestions`, `GET /agents`, `GET /messages`) that still use sync `requireTeamAccess` and may have the same staleness issue. These are explicitly deferred by scope, which is reasonable, but they should be tracked.

**Risk**: If a future refactor reverts the `usesPostgres` branch to an early-return pattern (e.g. while adding a cache flush optimisation), the board-wipe will silently resurface with no test to catch it.

**Recommendations**:
- Add a unit test for `persistTeamStore` that sets `teamStore.usesPostgres = true`, calls `persistTeamStore()`, and asserts `teamStore.set` was invoked for each team.
- Add an http-routes test that stubs `requireTeamAccessFresh` to return a fresh team different from the stale cached copy and asserts the `GET /board` response reflects the fresh data.
- File a follow-up task for the four remaining stale-read endpoints called out in the commit message.
