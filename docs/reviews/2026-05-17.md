This inconsistency means the `voice_error` payload carries an `unknown` value at runtime, which downstream event handlers would need to guard against. Compare with the parallel fix in `WebhookTrait.ts` and `TransformTrait.ts` in the same commit.


Recommended fix: `error: err instanceof Error ? err.message : String(err)`.

### 39f16187 — fix(mesh): add board task detail endpoint

**Author**: brianonbased-dev  
**Files changed**: `board-routes.ts` (+29 lines), `http-routes.test.ts` (+61 lines)

**Findings**

#### P3 — Double `pathname.match()` call is redundant

The new GET handler at line 597 runs `pathname.match(...)` to enter the branch, then immediately calls `pathname.match(...)` a second time at line 604 to capture the `taskId` group. The regex is identical in both calls. Since `pathname` is a string and `match()` is non-destructive, this is safe but wastes a second regex execution on every board-detail request.

```typescript
// guard (line 597)
if (pathname.match(/^\/api\/holomesh\/team\/[^/]+\/board\/[^/]+$/) && method === 'GET') {
  // ...
  // extract (line 604) — same regex again, just with a capture group
  const m = pathname.match(/^\/api\/holomesh\/team\/[^/]+\/board\/([^/]+)$/);
```

This mirrors how several other handlers in the same file work (e.g., the suggestions `/vote` handler at line 364) so it follows the local convention, but consolidating to one call would be cleaner.

#### P3 — `taskId` reflected in 404 response body without sanitisation

`json(res, 404, { error: 'Task not found', taskId })` echoes the caller-supplied `taskId` (after `decodeURIComponent`) verbatim. For a JSON API this is not an XSS risk, but it is an OWASP A03 (Injection) surface worth noting: if the `taskId` path segment ever contains embedded separators or control characters post-decode, they will appear in the response. At present `decodeURIComponent(m?.[1] || '')` is the only sanitisation. Recommend capping the echo to a fixed-length safe printable string, or omitting it from the 404 body.

**Positive observations**

- Authentication is correctly applied via `requireTeamAccessFresh(req, res, url, 'board:read')` before any data access. The team object is always scoped to the authenticated `teamId`, so there is no cross-team data leakage in the `doneLog` search.
- The non-null assertion `teamStore.get(teamId)!` is safe here: `requireTeamAccessFresh` populates `teamStore` as part of its fresh-load contract (consistent with every other handler in the same file).
- A test was included that covers both the happy path (task found) and the 404 path (missing task), including cross-checking `task.id` and `task.title` on the success response.

**Missing tests**: The test does not cover the `doneLog` path (i.e., look up a task that has been moved to done). This is a gap worth filing but not a blocker.

### f1fc5cef — fix(core): stabilize hardware validation gates

**Author**: brianonbased-dev  
**Files changed**: `holoMapMicroEncoder.ts`, `trait-commutativity.test.ts`, `MLSMPMFluid.ts`, `Phase9-11-NewSolvers.test.ts`

**Findings**

#### P2 — `weightCache` is an unbounded module-level `Map` with no eviction

`holoMapMicroEncoder.ts` introduces a module-level singleton `const weightCache = new Map<string, HoloMapMicroWeights>()`. Each distinct `(seed, modelHash)` pair allocates five `Float32Array` objects totalling `EMBED_DIM * (EMBED_DIM*4 + PATCH_LEN + 3)` floats — on the order of several hundred KB per entry depending on the constants. In a Studio session that cycles through many model configs (e.g., A/B asset comparison, or a batch encode job), this Map grows without bound and is never released until the module is unloaded.

The existing `holoMapWeightCache.ts` (IndexedDB layer, tested in `holoMapWeightCache.test.ts`) is a separate, disk-backed cache and does not apply here. The new in-memory Map has no corresponding test coverage of its cache-hit path or its unbounded growth.

Recommended fix: add an LRU cap (e.g., 16 entries) or clear on GPU device loss. At minimum, export a `clearMicroWeightCache()` for tests to call between runs, and add a test that verifies a second call with the same config returns the cached instance (referential equality).

#### P3 — `step(0)` in `Phase9-11-NewSolvers.test.ts` has an implicit semantic contract not documented in the test

Changing `solver.step(0.001)` to `solver.step(0)` causes `halfDtOverM = 0.5 * 0 / mass = 0`, so velocities are unaffected, positions are unaffected, but `computeForces()` is still called — giving a correct force/potential refresh at the exact placed position. This is the desired behavior for the test (verify PE at equilibrium without perturbing the system). However, `step(0)` also increments `stepCount` and calls `applyPBC()` and `berendsenThermostat(0)` — the thermostat divides by `dt` internally if `thermostatTau > 0`, which would produce `NaN` or `Infinity`. The test uses the default config (`thermostatTau` unset or zero), so there is no crash today, but this is a fragile assumption.

Recommend either: (a) call `computeForces()` directly (if it is accessible) to make the intent explicit, or (b) add a guard comment to the test noting the `thermostatTau: 0` assumption.

**Positive observations**

- The weight caching strategy itself is correct and well-motivated: before this change, every call to `createHoloMapMicroEncoder().run()` regenerated all five weight arrays from scratch using the same RNG seed, making them functionally identical but wasteful. The `microWeightKey` function correctly captures the two config fields that determine the weights.
- The `DeterminismHarness` removal in `trait-commutativity.test.ts` is correct: the harness was creating a new `ProvenanceSemiring` on every inner loop iteration (quadratic cost), and the replacement `hashBytes` call is both faster and simpler to read.
- The `MLSMPMFluid.ts` bind group fix (removing bindings 1–4 from the grid-clear group) aligns the CPU-side bind group layout with the WGSL shader's actual resource declarations. Binding 5 is the only slot the `cs_grid_clear` shader touches; the previous code was binding buffers that the shader ignores, which could cause validation errors on strict WebGPU implementations.

**Missing tests**: The `getMicroWeights` cache-hit path has no test. The `holoMapWeightCache.test.ts` file covers the IndexedDB layer, not the in-memory `Map`. A test that calls `getMicroWeights` twice with the same config and asserts referential equality (`toBe`) would confirm the cache works and provide a regression anchor.
