## P2 — Code Quality

### Q-01 · `b9edd28` · `verify-browser.ts` orphaned in CI

**File:** `packages/mcp-server/src/verify-browser.ts`

The commit message explicitly acknowledges this in passing ("verify-browser.ts is orphaned in CI" — tracked as `task_1778787124890_v819`), but it warrants a formal finding since it directly undermines the fix's durability.

`verify-browser.ts` is a complete, runnable browser smoke-test that launches Chromium, executes a script, takes a screenshot, and asserts success. It exists at `packages/mcp-server/src/verify-browser.ts` but is referenced by **no workflow** (checked `.github/workflows/`) and **no package.json script** in the `mcp-server` package. As a result:

- The runtime regression that this commit fixes (`"Executable doesn't exist at /usr/bin/chromium"`) could be reintroduced by any future change to `Dockerfile.mcp-server` without CI catching it.
- `self-healing-ci.yml` fires only on Railway deploy failures — it would catch the symptom (failed deploy), not the root cause (missing binary), and only after a production deploy attempt.

**Action:** Wire `verify-browser.ts` into CI. Minimal path:
1. Add `"verify-browser": "tsx src/verify-browser.ts"` to `packages/mcp-server/package.json` scripts
2. Add a step in `publish-mcp-image.yml` that runs `docker run --rm <image> pnpm --filter @holoscript/mcp-server verify-browser` after the image build step, gating on exit code
3. Close `task_1778787124890_v819` with the commit reference

## P3 — Operational / Style

### T-01 · `b9edd28` · Default image size increase affects all existing build paths

**File:** `infrastructure/Dockerfile.mcp-server`

Flipping `ARG INSTALL_CHROMIUM=0` → `1` is correct and necessary — without it, `browser_launch / browser_screenshot / browser_execute / hologram_render` were completely non-functional. The trade-off is a ~200 MB increase in the default image (Chromium + nss + freetype + harfbuzz + ttf-freefont via Alpine apk).

The escape hatch `--build-arg INSTALL_CHROMIUM=0` is preserved and documented in the updated comment. However, `publish-mcp-image.yml`, `railway.mcp-server.toml`, and `docker-compose.yml` pass no build-arg, meaning all three now implicitly build the heavier image. There is no slim-image CI job to verify that `INSTALL_CHROMIUM=0` still produces a working (non-browser) image.

**Action (low priority):** Consider adding a `SLIM_IMAGE` CI job in `publish-mcp-image.yml` that builds with `--build-arg INSTALL_CHROMIUM=0` and runs a non-browser MCP health check, confirming the opt-out path stays valid. Not urgent — the default-on choice is correct.
