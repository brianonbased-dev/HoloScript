### I-05 · `9506093e` · `defaultValidationReceipt` ignores `now` option — nondeterministic `validatedAt`

**File:** `packages/hololand-platform/src/creator/template-pipeline.ts:135`

The design contract states "Deterministic — same template + params + creatorId → same challenge id." `compileTemplateToChallenge` threads a `now` option through for the challenge timestamp, but `defaultValidationReceipt` ignores it and calls `new Date()` directly. Two calls with identical inputs at different milliseconds produce different receipts and different receipt hashes. Tests that inject `FIXED_NOW` will still get a varying `validatedAt`, making receipt hash assertions flaky. **Not fixed in `068fa131` — present in HEAD.**

**Action:** Pass `options.now` into `defaultValidationReceipt`: `validatedAt: options?.now ?? new Date().toISOString()`.

### I-09 · `9506093e` · `defaultValidationReceipt` uses djb2 as integrity hash

**File:** `packages/hololand-platform/src/creator/template-pipeline.ts:137–138`

`hash: djb2Hash(JSON.stringify(shard)), hashAlgorithm: 'djb2'` was introduced in `9506093e` and the hash algorithm was partially fixed to SHA-256 in `068fa131`. However, `djb2Hash` remains in the file for `buildChallengeId`, and the function comment says "replace with SHA-256 when a real backend needs collision resistance" — that backend now exists (the device-lab receipt pipeline). 32-bit djb2 has a birthday-bound collision at ~65k challenge IDs, making collisions realistic at scale if challenge IDs are used as unique persistent-store keys.

**Action:** Replace `djb2Hash` in `buildChallengeId` with `createHash('sha256').update(payload).digest('hex').slice(0, 16)` and remove the `djb2Hash` function.
