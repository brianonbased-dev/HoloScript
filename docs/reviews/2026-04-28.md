## Findings

### P1 — Broken Invariant

#### F-001 · Duplicate `OnnxMotionMatchingEngine` export collision
**Commits:** `33367f5` (feat(core): OnnxMotionMatchingEngine PFNN reference impl) and `3b99e08` (feat(traits): add OnnxMotionMatchingEngine with WebGPU/CPU/procedural fallback)
**Files:**
- `packages/core/src/traits/engines/onnx-motion-matching.ts` — class via `InferenceAdapter`; factory is **async**, throws if `infer()` called before `load()`
- `packages/core/src/traits/engines/motion-matching.ts` — class wrapping onnxruntime directly; factory is **sync**, silently falls back to procedural on `infer()`
- `packages/core/src/traits/engines/index.ts` — `export * from './motion-matching'` + `export * from './onnx-motion-matching'` → ambiguous re-export

**Conflicting names (verified):** `OnnxMotionMatchingEngine`, `ModelDescriptor`, `BUNDLED_MODELS`, `createOnnxMotionMatchingEngine`

TypeScript does not raise an error on ambiguous `export *` re-exports at the barrel level — the conflict is silent until a consumer imports one of the colliding names, at which point the bundler picks whichever file it resolves first. The two implementations have opposite contracts (throws vs. graceful-fallback) so the wrong pick causes silent behavioral divergence.

**Required action:** One implementation must be the canonical one. The `onnx-motion-matching.ts` version is the more complete (tests cover encoding/decoding, batch inference, error paths). The additions to `motion-matching.ts` from `3b99e08` should either be removed or renamed and re-scoped. Until resolved, the barrel export is broken.

#### F-003 · Regex XML parsing in `urdf-plugin`
**Commit:** `9411244`
**File:** `packages/plugins/urdf/src/index.ts:55–130`

`importFromUrdf` parses URDF XML with hand-rolled regexes. The comment says "Uses regex-based extraction for portability (no DOM dependency)." The fragility surface:

- The geometry-lookup regex `/<link[^>]+name="${name}"[\s\S]*?<geometry>[\s\S]*?<(box|cylinder|sphere|mesh)/` scans the entire document from link-name forward, so the geometry of *link B* is attributed to *link A* if A's geometry block appears after B's in document order.
- The mass regex has the same greedy-scan problem.
- Any attribute value containing `"` (escaped as `&quot;` in XML) or a link name containing regex metacharacters will silently break the match.
- The 14 existing tests all use a controlled fixture; no adversarial or out-of-order XML is exercised.

This is not a F.014 violation (F.014 covers `.hs`/`.hsplus` outside `@holoscript/core`), but regex XML parsing is inherently fragile. Use `@xmldom/xmldom` (already present elsewhere in the monorepo at `packages/core`) or the native `DOMParser` API.

#### F-004 · `any` types in new parser test code
**Commit:** `30e4a95` (fix(parser): whitelist stress-test directives)
**File:** `packages/core/src/parser/HoloScriptPlusParser.test.ts`

Lines 133, 166, 173, 174 use `(c: any)`, `(w: any)`, `(d: any)` inline casts. CLAUDE.md style: *prefer `unknown` over `any`*. The AST node types are exported from `@holoscript/core` — the test should import them or use `unknown` with a type guard. Using `any` here defeats type-checking in the test assertions.

```ts
// current (line 166)
const unknownWarnings = (result.warnings || []).filter((w: any) =>

// preferred
const unknownWarnings = (result.warnings || []).filter((w: unknown) =>
  typeof (w as { message?: unknown }).message === 'string' &&
  (w as { message: string }).message.includes('Unknown directive')
);
```

#### F-005 · `as string` cast papers over Express 5 `req.params` type
**Commit:** `a3ec96a` (fix(ci): unblock CI — llm-service TS2345)
**File:** `services/llm-service/src/server.ts` lines 246, 258

The fix casts `req.params['id'] as string` to silence TS2345. Express 5's `ParamsDictionary` types `params` as `string | string[]`. The cast is unsafe — if the router ever matched a repeated parameter, the string-typed downstream call (`buildService.getBuild`, `buildService.deleteBuild`) would receive `"[object Array]"`.

```ts
// safer fix
const id = Array.isArray(req.params['id']) ? req.params['id'][0]! : req.params['id'];
```

The same file also uses `(req as any).apiKey` (pre-existing) — the `apiKey` property should be typed via Express `Request` declaration merging rather than `any` cast.

#### F-006 · `infer()` silently ignores loaded ONNX session
**Commit:** `3b99e08`
**File:** `packages/core/src/traits/engines/motion-matching.ts:483–490`

The synchronous `infer()` method on the newly added `OnnxMotionMatchingEngine` always calls `runProceduralInference(input)` — even when `this.session` is set and `this.activeBackend === 'webgpu'`:

```ts
infer(input: MotionInferenceInput): MotionInferenceResult {
  if (this.session && this.ort) {
    // ...comment about onnxruntime not supporting sync...
    return runProceduralInference(input);   // always procedural
  }
  return runProceduralInference(input);
}
```

Both branches are identical. A caller checking `engine.backend === 'webgpu'` reasonably expects neural inference; they silently get procedural output. The dead `if (this.session && this.ort)` branch should either be removed or the method should throw `UnsupportedOperationError` directing callers to `inferAsync()`. This is also moot if F-001 is resolved by removing this file's `OnnxMotionMatchingEngine` entirely.
