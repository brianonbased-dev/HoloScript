## New Findings

### P1 — Broken Invariant

#### F-009 · F-008 fix has bash operator-precedence bug — JS failures still silenced
**Commit:** `b7b1c16`
**File:** `packages/studio/Dockerfile` lines 60–86

The fix replaced `npx tsup || true` with `npx tsup --no-dts && npx tsup --dts-only || true`.
In bash, `&&` and `||` are left-associative with equal precedence, so the shell evaluates:

```
(cd pkg && npx tsup --no-dts && npx tsup --dts-only) || true
```

If `npx tsup --no-dts` exits non-zero (a real JS compilation failure), the `&&` short-circuits the whole left-hand side to false, then `|| true` fires and the `RUN` layer exits 0 — Docker build continues with an empty `dist/`. This is identical in effect to the original F-008 bug.

The correct pattern (braces isolate `|| true` to `--dts-only` only):

```dockerfile
RUN cd packages/crdt && npx tsup --no-dts && { npx tsup --dts-only || true; }
```

**Action required:** fix all 19 affected `RUN` lines in `packages/studio/Dockerfile`.

### P2 — Quality / Correctness

#### F-011 · F.014 violation — regex `.holo` parsing outside `@holoscript/core`
**Commit:** `829be76`
**File:** `packages/studio/src/app/dev/ui-graph/page.tsx:57–85`

`parseHolo()` uses a series of regexes (including `pageBlockRegex = /^ {2}page (\w+) \{…/gm`) to parse the `.holo` artifact directly in a Next.js server component. Per CLAUDE.md behavioral override (F.014):

> NEVER hand-write .hs/.hsplus/.holo parsing with regex outside @holoscript/core

This creates a second `.holo` parser that can drift from the canonical one and will not catch spec changes. The core's `parse_holo` / AST walk should be used instead.

#### F-012 · Runtime path bug — `HOLO_PATH` resolves incorrectly in production
**Commit:** `829be76`
**File:** `packages/studio/src/app/dev/ui-graph/page.tsx:38`

```typescript
const HOLO_PATH = join(process.cwd(), '.holo', 'studio.ui.holo');
```

On Railway, `process.cwd()` is `/app` (the Docker workdir). The `.holo` artifact lives at `packages/studio/.holo/studio.ui.holo` — i.e., `/app/packages/studio/.holo/studio.ui.holo`. The constructed path `/app/.holo/studio.ui.holo` does not exist; every request to `/dev/ui-graph` in production returns the "UI Graph not generated" error branch. The commit message explicitly notes "visual rendering not yet verified."

**Suggested fix:** anchor relative to `import.meta.url` or use a `STUDIO_ROOT` env var set in the Dockerfile.

#### F-013 · Unauthenticated write to embodied status cache — CORS wildcard on POST
**Commit:** `3070548`
**File:** `packages/studio/src/app/api/embodied/composite/route.ts`

When `EMBODIED_NOTIFY_KEY` is not set (the default for a fresh Railway deploy), any actor can `POST /api/embodied/composite` and inject arbitrary system status into Brittney's `read_embodied_status` tool responses. CORS is `Access-Control-Allow-Origin: *` on both GET and POST. The comment calls this "dev mode" but the code ships to the same production container as authenticated routes.

Severity: P2 rather than P0 because the injected data is non-privileged (no exfiltration, no command execution), but an attacker could mislead Brittney into reporting false system health.

**Recommended fix:** require Bearer token in `NODE_ENV === 'production'` regardless of whether `EMBODIED_NOTIFY_KEY` is set; refuse startup without it in prod.
