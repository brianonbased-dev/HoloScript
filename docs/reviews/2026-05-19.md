## Findings

### P2 - `holo_tunnel_create` can expose arbitrary local or internal hosts

**Commit**: `9130117ad` - `test(mcp): prove holotunnel lifecycle`
**Files**:

- `packages/mcp-server/src/holo-tunnel-tools.ts`
- `packages/mcp-server/src/__tests__/holo-tunnel-tools.test.ts`

The HoloTunnel MCP tool exposes `relayBase` and `localHost` as user-supplied arguments, then passes them directly to `startHoloTunnel`:

- `packages/mcp-server/src/holo-tunnel-tools.ts:106` defines `relayBase` as an optional input.
- `packages/mcp-server/src/holo-tunnel-tools.ts:111` defines `localHost` as an optional input.
- `packages/mcp-server/src/holo-tunnel-tools.ts:210` reads `localHost` with no hostname allowlist.
- `packages/mcp-server/src/holo-tunnel-tools.ts:211` reads `relayBase` with no relay allowlist.
- `packages/mcp-server/src/holo-tunnel-tools.ts:218` passes both values into `startHoloTunnel`.

The test added in the same commit proves the passthrough behavior by sending `localHost: '127.0.0.1'` and an explicit production relay, then asserting those exact values were forwarded (`packages/mcp-server/src/__tests__/holo-tunnel-tools.test.ts:88` through `:128`).

That is fine for a trusted local CLI, but this surface is an MCP tool. An agent with tool access can ask the host to publish more than the intended local dev server: another loopback service, a LAN address, or an attacker-controlled relay URL if `relayBase` is accepted in production. HoloTunnel is exactly the right direction for Quest-without-cable workflows, but the boundary needs to be crisp before it becomes the default device bridge.

**Recommended fix**: Keep the override path for local testing, but gate it. Default production behavior should only permit `localhost` or `127.0.0.1` plus the expected production relay. Require an explicit dev/test environment flag for arbitrary `localHost` and `relayBase`, and add negative tests for private-network hosts and unapproved relays.

### P2 - PR-only security checks became unreachable after workflow-dispatch migration

**Commit**: `1b159111a` - `ci: move holoscript validation in house`
**Files**:

- `.github/workflows/ci.yml`
- `.github/workflows/security.yml`
- `docs/ops/in-house-validation.md`

The in-house validation direction is documented clearly: `docs/ops/in-house-validation.md:37` says workflows are intentionally `workflow_dispatch` only, and lines `:47` through `:52` define the expected receipt fields. The issue is enforcement. Critical workflows now start with only:

- `.github/workflows/ci.yml:3` `on:`
- `.github/workflows/ci.yml:4` `workflow_dispatch:`
- `.github/workflows/security.yml:3` `on:`
- `.github/workflows/security.yml:4` `workflow_dispatch:`

Inside `security.yml`, the dependency-review job is still guarded by `if: github.event_name == 'pull_request'` at line `:48`. Because the workflow no longer listens to `pull_request`, that job is effectively dead. The same pattern affects any branch-protection expectation that used GitHub check runs as the machine-readable proof of lint, tests, CodeQL, dependency review, docs quality, or package builds.

The new operating model can still be valid, but it needs a machine-enforced replacement for the removed automatic checks. A handoff note that says "run the narrowest command set" is useful for humans and agents, but it is not a branch-protection gate and does not stop an unreceipted merge.

**Recommended fix**: Either restore lightweight `pull_request` triggers for the essential gates, or add a required HoloMesh/local-receipt status check that verifies commit SHA, command list, pass/fail summary, and artifact paths before merge. If the GitHub workflows are deliberately manual-only, remove or rewrite dead `github.event_name == 'pull_request'` jobs so the repository does not advertise checks that cannot run.
