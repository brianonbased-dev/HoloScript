#### F-1 · Simulated benchmark data presented as real measurements
**Commit**: `27e81302`
**File**: `scripts/run-emergent-spacetime-benchmark.mjs` (lines 99–115 of the script)

The runner script does **not** execute the TypeScript benchmark harness. It outputs a `simulatedResult` object with hand-crafted constants (`avg500Voxel: 12.5`, `avg1000Voxel: 24.8`, `avgRicciPerVoxel: 5.2`) that always pass the Paper 3 §7.8 claims, regardless of actual hardware or code behavior. The script says:

```js
// Note: In a real run, we would import and execute the TypeScript benchmark.
// For now, this script provides the CLI interface and will be wired up
// after the TypeScript is compiled.
```

The problem: the verification summary it prints reads `ALL CLAIMS VERIFIED ✓` for every invocation with no qualification. Any CI job or researcher running this script will see a green result backed by invented numbers.

**Required fix**: Either (a) gate the script so it exits non-zero with a clear "STUB — not wired to real harness" message, or (b) wire it to actually compile and invoke the TypeScript harness before producing output. The current state is unusable for Paper 3.

#### F-2 · `avgRicciPerVoxel` not normalized by voxel count
**Commit**: `27e81302`
**File**: `packages/studio/src/__benchmarks__/emergent-spacetime-rtx-benchmark.ts` (lines ~155–162)

```typescript
const avgRicciPerVoxel = runs.reduce((sum, r) => sum + r.ricciComputationTime, 0) / runs.length;
```

`ricciComputationTime` is the **total** μs for a single Ricci call across all voxels in that run. `avgRicciPerVoxel` therefore averages this total across voxel-count sizes (500, 1000, 2000) and calls the result "per voxel". It never divides by voxel count.

If the Ricci call takes 5 μs for 500 voxels and 10 μs for 1000 voxels, `avgRicciPerVoxel` = 7.5 μs — not 0.01 or 0.01 μs/voxel. The Paper 3 claim is "<10 μs/voxel"; the benchmark is effectively measuring total-μs-averaged-over-sizes, not per-voxel cost.

**Required fix**: Compute `ricciComputationTime / voxelCount` in `BenchmarkMetrics` (or in the summary calculation) to produce a genuine per-voxel figure.

#### F-7 · `GET /api/hologram/share` leaks `createdBy` without authentication
**Commit**: `4256a798`
**File**: `packages/studio/src/app/api/hologram/share/route.ts` (lines ~826–838)

The `POST` handler runs `authorizeHologramUpload` before acting. The `GET` handler has **no auth check** and returns the `createdBy` field (which is a user identifier — e.g., `user@example.com` for authenticated Studio sessions) to any unauthenticated caller who knows a bundle hash.

```typescript
return NextResponse.json({
  hash: record.hash,
  url: `/g/${record.hash}`,
  shared: true,
  expired,
  createdAt: record.createdAt,
  expiresAt: record.expiresAt,
  viewCount: record.viewCount,
  createdBy: record.createdBy,   // ← exposed to unauthenticated callers
});
```

**Required fix**: Either strip `createdBy` from the public GET response, or gate the full record behind the same `authorizeHologramUpload` check used by POST.

#### F-8 · `/gram/[hash]/page.tsx` skips hash validation before `redirect()`
**Commit**: `4256a798`
**File**: `packages/studio/src/app/gram/[hash]/page.tsx`

```tsx
export default async function GramAliasPage({ params }) {
  const { hash } = await params;
  redirect(`/g/${hash}`);   // no assertValidHash call
}
```

Every other hash-handling entry point in this codebase calls `assertValidHash` before doing anything with the hash. This route passes the raw Next.js param directly to `redirect()`. While same-origin redirects prevent open-redirect to external URLs, the inconsistency with the project's security pattern is a bug waiting to happen (e.g., if `redirect` behavior changes or the route is refactored).

**Required fix**: Add `assertValidHash(hash)` before the `redirect()` call, with a `notFound()` catch for invalid hashes.
