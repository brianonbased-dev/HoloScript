### F-001 — P0 — Security: API token exposed via subprocess CLI argument

**Commit:** `128e9657` (`feat(quantum): IBM Quantum integration`)
**File:** `packages/plugins/qm-bridge/src/backends/ibm-quantum.ts`

The `IBMQuantumBackend._runPythonBridge()` method constructs the input payload as:

```ts
const input = {
  ...moleculeOrGraph,
  api_token: this._resolveApiToken(),   // ← IBM Quantum API token
  execution_mode: this.config.executionMode ?? 'aer',
};

const { stdout, stderr } = await execFileAsync(
  'python3',
  [scriptPath, JSON.stringify(input)],   // ← entire JSON incl. token on argv
  { maxBuffer: 10 * 1024 * 1024 },
);
```

`JSON.stringify(input)` is passed as a positional CLI argument. On any Linux/macOS system, every process's argv is readable by any user via `ps aux` or `/proc/<pid>/cmdline`. The IBM Quantum API token is therefore visible to all local users while the subprocess runs.

**Fix:** Remove `api_token` from the JSON payload and pass it via the subprocess environment instead:

```ts
const { stdout, stderr } = await execFileAsync(
  'python3',
  [scriptPath, JSON.stringify(inputWithoutToken)],
  {
    maxBuffer: 10 * 1024 * 1024,
    env: {
      ...process.env,
      IBM_QUANTUM_API_KEY: this._resolveApiToken() ?? '',
    },
  },
);
```

The Python script already reads `IBM_QUANTUM_API_KEY` from the environment as a fallback, so this is a low-friction fix.

### F-002 — P1 — Style: Implicit `any` in PuppeteerRenderer label helpers

**Commit:** `4bd40952` (`fix(cli): render object labels in screenshots`)
**File:** `packages/engine/src/rendering/headless/PuppeteerRenderer.ts`

Two helper functions are defined without TypeScript parameter annotations:

```ts
function createLabelSprite(text, accentColor) { ... }
function roundedRect(ctx, x, y, width, height, radius) { ... }
```

With `noImplicitAny: true` (which this monorepo enforces via strict tsconfig), these will fail compilation. Even if they live inside a Puppeteer `evaluate()` serialized closure, they are still parsed and type-checked by `tsc`. If tests are currently green it suggests tsconfig strict mode is not fully propagated to `@holoscript/engine` — which is itself worth confirming.

**Fix:**

```ts
function createLabelSprite(text: string, accentColor: string): THREE.Sprite { ... }
function roundedRect(
  ctx: CanvasRenderingContext2D,
  x: number, y: number,
  width: number, height: number,
  radius: number,
): void { ... }
```

### F-003 — P1 — Missing test guards: connector-appstore CI failures

**Commit:** `769a6b48` (`feat(training+scripts): Brittney GRPO training pipeline + utility scripts`)
**File:** `packages/connector-appstore/IMPLEMENTATION_SUMMARY.md` (documents the issue), source tests

The implementation summary self-reports:

> **Passing:** 38 (63%)
> **Failing:** 22 (37% — require real API credentials)

The 22 failing tests (`AppleAppStoreClient` and partial `GooglePlayClient` suites) are left as hard failures rather than being skipped with `it.skipIf(!process.env.APPLE_KEY_ID, ...)` or similar. CI will report a red test suite for this package on every run where credentials are absent — which is every standard CI run.

**Fix:** Guard credential-dependent tests:

```ts
const hasAppleCreds = Boolean(process.env.APPLE_KEY_ID && process.env.APPLE_ISSUER_ID);
it.skipIf(!hasAppleCreds)('generates JWT token', async () => { ... });
```

**File:** `packages/studio/public/wasm/holoscript.js` (8,920 lines)

### F-007 — P2 — Code quality: fragile six-level relative path for Python script

**Commit:** `128e9657`
**File:** `packages/plugins/qm-bridge/src/backends/ibm-quantum.ts`

```ts
const scriptPath =
  this.config.scriptPath ??
  new URL('../../../../../../scripts/quantum_execute.py', import.meta.url).pathname;
```

Six levels of `..` is brittle. If the package moves or the monorepo layout changes, this will silently resolve to a wrong path at runtime (no error until the subprocess fails). Consider anchoring to a workspace root resolved at startup or at least asserting the resolved path exists before use.

### F-010 — P3 — Minor: Stripe webhook test script uses plaintext fallback secret

**Commit:** `769a6b48`
**File:** `services/absorb-service/scripts/test-stripe-webhook.js`

```js
const webhookSecret = process.env.STRIPE_WEBHOOK_SECRET || 'whsec_test_secret';
```

This is a dev-only test script and the fallback is clearly labeled as a test value. The risk is low. However, if this script ever runs against a staging endpoint with a real secret injected at runtime, the fallback masks a missing environment variable silently. Prefer a hard failure:

```js
const webhookSecret = process.env.STRIPE_WEBHOOK_SECRET;
if (!webhookSecret) throw new Error('STRIPE_WEBHOOK_SECRET is required');
```
