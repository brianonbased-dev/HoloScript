## New Findings

### P2 — Quality / Correctness

#### F-008 · `|| true` silences all tsup failures in studio Dockerfile
**Commit:** `3d2980c` (fix(studio,dockerfile): comprehensive sweep)
**File:** `packages/studio/Dockerfile`

Commit `3d2980c` replaced `npx tsup --no-dts` with `npx tsup || true` for
approximately 18 package build steps. The stated intent is to tolerate dts
emission failures while still surfacing type gaps through the downstream
studio tsc step. However, `npx tsup || true` swallows *all* tsup exit
codes, including JavaScript compilation failures — not just dts failures.

If a package's JS build fails (e.g., a syntax error in `packages/crdt`,
`packages/mesh`, or `packages/llm-provider`), the Docker layer succeeds with
an empty or partial `dist/` folder. The studio container then deploys silently
broken, and the failure only surfaces at runtime as a `Cannot find module`
500 error — the same class of error the fix was meant to eliminate.

**Confirmed at-risk layers:**

```dockerfile
RUN cd packages/crdt && npx tsup || true          # foundational dep
RUN cd packages/mesh && npx tsup || true          # studio networking dep
RUN cd packages/llm-provider && npx tsup || true  # Brittney route dep
```

**Recommended fix:** Separate the JS and dts steps so only dts failures are
tolerated:

```dockerfile
RUN cd packages/crdt && npx tsup --no-dts && npx tsup --dts-only || true
```

Or, if tsup does not support `--dts-only`, use the two-call pattern:

```dockerfile
RUN cd packages/crdt && npx tsup --no-dts  # JS build must succeed
RUN cd packages/crdt && npx tsup || true   # dts pass may fail
```

The two packages that intentionally skip dts (`core` via `generate-types.mjs`,
`engine` via `build-engine-no-dts.sh`) are correctly handled and not affected
by this finding.

### P3 — Minor / Style

#### F-009 · Indentation drift in `VRTraitSystem.ts` from `@ts-expect-error` strip
**Commit:** `42da5e4` (fix(core/traits/VRTraitSystem): clear typecheck slice)
**File:** `packages/core/src/traits/VRTraitSystem.ts`

When `@ts-expect-error` directives were mechanically removed in `42da5e4`,
the subsequent code lines were left with extra indentation relative to their
surrounding blocks. Three confirmed sites remain in the current file after
the follow-up `1ffaf90` (which fixed the `rotate_start`/`onUpdate` sites but
did not touch these):

**Site 1 — `throwable` handler (line 621):**
```ts
// Surrounding block: 12-space indent
const multiplier = ...;         // 12 spaces ✓
                    context.physics.applyVelocity(node, [  // 20 spaces ✗
  velocity[0] * multiplier,      // 14 spaces (inside the call)
```
The `applyVelocity` call has 8 extra spaces; its array elements are *less*
indented than the call itself.

**Site 2 — `rotatable` switch-case (lines 987–1000):**
```ts
      case 'x':                  // 6 spaces
            newRotation = [...]; // 12 spaces ✗ (was 8, pre-edit)
        break;                   // 8 spaces ✓
```
`newRotation` assignments sit 4 spaces deeper than the `break` they precede.
Same pattern in `case 'y'`, `case 'z'`, and `default`.

**Site 3 — `snappable` snap-angle guard (line 1005):**
```ts
    if (config.snap_angles ...) {  // 4 spaces
        newRotation = newRotation.map(...); // 8 spaces ✗ (was 6)
```

None of these affect runtime behavior, but they will cause noise in ESLint /
Prettier CI and make the code harder to read. A single formatting pass over
the three sites would resolve all of them.

#### F-010 · Multi-paragraph module docstring in `anthropic-streaming.test.ts`
**Commit:** `ce9c6d9` (feat(llm-provider): unified streamCompletion surface)
**File:** `packages/llm-provider/src/__tests__/anthropic-streaming.test.ts:1–17`

The new test file opens with a 17-line multi-paragraph module-level comment
block. CLAUDE.md style: *"Never write multi-paragraph docstrings or multi-line
comment blocks — one short line max."* The docstring explains the D.025 context
and four contract invariants being pinned. While the content is useful, the
contract invariants belong in the `describe`/`it` names, and the D.025 context
belongs in the PR description or a short in-test comment. The test file also
has a per-test `// Why this is wired up` multi-line block inside one test
(lines after the `import` group).

**Fix:** Replace the module docstring with a single line:
```ts
// Pins AnthropicAdapter.streamCompletion chunk translation — D.025 streaming surface contract.
```
Move the four invariant descriptions into the `it(...)` names or inline `//`
comments above each assertion group.
