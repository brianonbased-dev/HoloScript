**Findings**

#### P2 ‚Äî Color mismatch between raw `emissive` property and `@emissive` trait on `scan_ring_3km`

In `examples/quantum-radar.refreshed.hs`, the object `scan_ring_3km` sets two conflicting emissive colors:

```holo
// scan_ring_3km (line ~30 of quantum-radar.refreshed.hs)
material: { baseColor: "#00ffff", roughness: 0.1, metallic: 0.0 }
emissive: "#00ffff"          // raw property ‚Äî cyan
emissiveIntensity: 1.2

@emissive { intensity: 1.2, color: "#00ccff" }   // trait config ‚Äî slightly blue-shifted
```

The raw `emissive` property is `#00ffff` (pure cyan, RGB 0/255/255) while the `@emissive` trait's `color` is `#00ccff` (blue-shifted, RGB 0/204/255). Depending on runtime property/trait merge precedence ‚Äî which is not documented for this conflict case ‚Äî the ring will either render cyan or a perceptibly different steel blue. The artist's intent (matching the radar sweep aesthetic) cannot be determined without knowing the intended color.

By contrast, `scan_ring_1km` sets both to `#00ff88` (identical values ‚Äî intentional duplication), and `scan_ring_7km` omits the `@emissive` trait entirely.

The mismatch is not a parser error (the file compiled cleanly), but it is a latent visual bug and an authoring anti-pattern: raw material properties and trait configs for the same attribute should not disagree.

**Recommended fix**: Decide the intended color and make both consistent ‚Äî or remove the raw `emissive`/`emissiveIntensity` properties and rely solely on `@emissive` (the preferred trait-first approach).

#### P3 ‚Äî `docs/examples-health/2026-05-17.md` added without a VitePress sidebar entry (recurring)

Same finding as flagged in the 2026-05-17 review for `docs/examples-health/2026-05-16.md`. The `docs/examples-health/` folder has 12 date-stamped files (dating back to 2026-04-28) with zero entries in `docs/.vitepress/config.ts`. Either the folder is intentionally excluded from the user-facing docs (in which case this should be documented), or the sidebar updates have been systematically skipped since the folder was created.

This has now been raised in two consecutive daily reviews. **Recommended action**: decide the folder's status (internal audit log vs. user-facing docs) and document it in a `docs/examples-health/README.md` or in the VitePress config comment. If user-facing, add a single top-level sidebar entry for the folder rather than registering each date file individually.
