## Detailed Findings

### R001 — P1 — F.014 violation: hand-rolled `.hsplus` lexer outside `@holoscript/core`

**Commit:** 6b4156c (`feat(mesh): add holomesh agent control tools`)
**File:** `packages/mcp-server/src/holomesh/agent-capability-tools.ts`

The `holomesh_agent_capabilities` MCP tool ships a 200-line custom tokenizer/lexer
(`lexMetadataTokens`, `extractStringArrayBlock`, `extractDecoratorTraits`,
`extractCapabilityBlockNames`, `collectStringsUntilBalanced`) that parses `.hsplus`
source to extract trait names, capability tags, and composition structure. Only the
`extractCompositionName` function delegates to `parseHoloPartial` from
`@holoscript/core`; all other extraction goes through the bespoke lexer.

This violates anti-pattern F.014 ("regex/manual parsing of `.hs/.hsplus` outside
`@holoscript/core`"). The invariant exists because the canonical parser enforces
semantic rules (inheritance, trait validation, operator precedence) that a hand-rolled
tokenizer cannot guarantee. If the `.hsplus` grammar evolves, the bespoke lexer will
silently drift and produce wrong capability summaries for agent brains.

**Fix direction:** Extend `parseHoloPartial` in `@holoscript/core` to expose a
`capabilityMeta()` extraction helper. Then delegate all block extraction to that API
from `agent-capability-tools.ts`. The bespoke `lexMetadataTokens` function should be
deleted once the core helper covers the same fields.

### R002 — P2 — No rate-limiting on unauthenticated `/oauth/github/device`

**Commit:** 688eaa9 (`feat(auth): GitHub OAuth 2.0 device authorization grant on HoloMesh`)
**File:** `packages/mcp-server/src/http-server.ts`, `packages/mcp-server/src/security/github-device-flow.ts`

The new `POST /oauth/github/device { action: 'initiate' }` path is publicly
accessible (no Bearer token required), calls GitHub's `POST /login/device/code`
endpoint on every request, and has no per-IP, per-session-count, or per-minute limit.

An unauthenticated caller can flood this endpoint and exhaust the GitHub OAuth app's
rate limit for `GITHUB_DEVICE_FLOW_CLIENT_ID`, blocking legitimate device-flow auth
for all HoloMesh users.

**Fix direction:** Either require authentication on this endpoint (callers who want to
start a device flow must already have a partial-trust bearer) or add per-IP rate
limiting (e.g. max 3 `initiate` calls per IP per 15-minute window) before calling
GitHub. A simple in-process leaky-bucket per-IP map is sufficient given the existing
architecture.

### R003 — P2 — `setInterval` at module scope leaks into test suites

**Commit:** 688eaa9 (`feat(auth): GitHub OAuth 2.0 device authorization grant on HoloMesh`)
**File:** `packages/mcp-server/src/security/github-device-flow.ts:92`

```typescript
setInterval(cleanupExpiredSessions, CLEANUP_INTERVAL_MS);  // runs on import
```

This fires unconditionally when the module is imported. Any test file that directly or
transitively imports this module inherits a 60-second background timer. In Vitest's
pool executor this can prevent the worker from exiting cleanly, causing spurious test
timeouts or requiring `--forceExit`.

The existing test file (`security/__tests__/github-device-flow.test.ts`) avoids the
issue today because it mocks `fetch`, but any future test that imports the handlers
module transitively will be affected.

**Fix direction:** Wrap the `setInterval` call in a lazy initialiser or export a
`startCleanup() / stopCleanup()` pair. The `startCleanup()` should be called from
`http-server.ts` at startup, not at module load time.

### R004 — P2 — In-memory session store lost on Railway redeploy

**Commit:** 688eaa9 (`feat(auth): GitHub OAuth 2.0 device authorization grant on HoloMesh`)
**File:** `packages/mcp-server/src/security/github-device-flow.ts:70-72`

```typescript
// ── In-memory store (swap for Redis/Postgres in production) ───────────────
const sessions = new Map<string, DeviceFlowSession>();
```

The code comment acknowledges this is not production-ready, but HoloMesh runs on
Railway with no persistent memory between deploys. Any active device-flow session
(the user is looking at the `user_code` screen on their phone) will silently vanish
on the next deploy. The user sees a confusing "authorization pending" error with no
explanation.

Given the 15-minute TTL this is tolerable for the MVP, but the in-memory state should
be tracked as a known production gap. The self-healing CI workflow fires on deploy
failures and could trigger at any time while a device flow is in progress.

**Fix direction:** File a board task for Redis/persistent-store migration, or add a
telemetry event when a session is looked up and not found so the gap is observable.
No immediate code change required if the team accepts this trade-off consciously.

### R005 — P2 — `action=stats` diagnostic endpoint is unauthenticated

**Commit:** 688eaa9 (`feat(auth): GitHub OAuth 2.0 device authorization grant on HoloMesh`)
**File:** `packages/mcp-server/src/http-server.ts`

The `POST /oauth/github/device { action: 'stats' }` path returns:
- total sessions initiated
- sessions currently pending
- sessions complete / expired

…to any caller without a Bearer token. While this is low-sensitivity data, it leaks
information about platform usage patterns to unauthenticated parties.

**Fix direction:** Move the `stats` action behind at least a founder-only auth check,
or gate it on the existing `FOUNDER_API_KEY` header check used elsewhere in
`http-server.ts`.

### R006 — P3 — `path.resolve` without directory boundary in `holomesh_agent_capabilities`

**Commit:** 6b4156c
**File:** `packages/mcp-server/src/holomesh/agent-capability-tools.ts:357`

```typescript
const resolved = path.resolve(brainPath);
// No check: resolved.startsWith(ALLOWED_ROOT)
```

When `allow_file_read=true`, the `brain_path` argument is resolved to an absolute path
and read without verifying it falls within an allowed directory. A caller (including
an agent acting on behalf of another) can read any file accessible to the server
process (e.g. `../../.env`, `../../../secrets`). The `max_bytes` guard limits data
volume but not access scope.

**Fix direction:** After `path.resolve`, check that the resolved path starts with a
configured `BRAIN_FILE_ALLOWED_ROOT` (defaulting to `process.cwd()`). Reject with
`403` if outside the boundary.

### R007 — P3 — `ALLOWED_MOBILE_SCOPES === DEFAULT_MOBILE_SCOPES` makes scope narrowing a no-op

**Commit:** 69299b2 (`feat(mobile-handoff): POST /api/holomesh/team/:id/mobile-handoff endpoint`)
**File:** `packages/mcp-server/src/holomesh/routes/team-routes.ts:838-844`

```typescript
const DEFAULT_MOBILE_SCOPES = ['holomesh:read', 'team:read', 'team:message'];
const ALLOWED_MOBILE_SCOPES = new Set(DEFAULT_MOBILE_SCOPES);       // same set
const scopedSubset = requestedScopes.filter((s) => ALLOWED_MOBILE_SCOPES.has(s));
const scopes = scopedSubset.length > 0 ? scopedSubset : DEFAULT_MOBILE_SCOPES;
```

A caller requesting `['holomesh:read']` (intending a narrower mobile token) would
receive `['holomesh:read']` — which is correct — but requesting any unrecognised scope
silently falls back to the full `DEFAULT_MOBILE_SCOPES`. The code comment promises
"allow parent to request a subset of their own effective permissions," but there is no
check that requested scopes are a subset of the parent caller's actual `KeyRecord.scopes`.
A caller with only `holomesh:read` can produce a mobile token with `team:message`
(which may not match their intent).

This is not an escalation vulnerability (the mobile token cannot exceed
`ALLOWED_MOBILE_SCOPES`) but does mean the parent cannot actually narrow the token
below the default.

**Fix direction:** If scope narrowing is desired, filter against the intersection of
`ALLOWED_MOBILE_SCOPES` and the caller's own `record.scopes`. If it isn't needed yet,
document that the `scopes` request body field is accepted but currently ignored.
