### Q-01 · `14c7fb90` · No regression test for fresnel fix

**File:** `packages/studio/src/features/shader-editor/LivePreviewService.ts`

The fix removes `fresnel: 'CustomGLSL'` from `mapShaderNodeType`, restoring the intended path where bare `'fresnel'` passes through to identity and WGSLTranslator handles it natively. The fix is correct and well-described. However, no test was added that would have caught the original regression (`ab9d41c1`) or would prevent re-introduction of the alias. The fresnel → CustomGLSL bug existed from `ab9d41c1` until this commit — a span in which any shader using a fresnel node silently produced unchanged-input output.

**Action:** Add a unit test in the shader-editor test suite that:
1. Creates a `ShaderGraph` node with type `'fresnel'`
2. Calls `mapShaderNodeType('fresnel')`
3. Asserts the result is `'fresnel'` (identity), not `'CustomGLSL'`
4. Asserts WGSLTranslator produces the `pow(1-max(dot(view,normal),0), power)` expression

### Q-02 · carry-over from I-03 (2026-05-08) · `utility_fresnel` absent from WGSLTranslator

**Context:** I-03 in yesterday's review flagged two distinct issues:

1. `fresnel: 'CustomGLSL'` incorrect alias → **fixed in `14c7fb90`** ✓
2. `utility_fresnel` falls through to identity in `mapShaderNodeType`, and WGSLTranslator has no handler for `'utility_fresnel'` → **not addressed in any commit today**

**File:** `packages/studio/src/features/shader-editor/LivePreviewService.ts` (mapShaderNodeType), `packages/studio/src/features/shader-editor/WGSLTranslator.ts`

`MaterialLibrary.ts` and `ShaderTemplates.ts` use `ShaderGraph.createNode('utility_fresnel', ...)`. With `fresnel: 'CustomGLSL'` now removed, bare `'fresnel'` works. But `'utility_fresnel'` still returns from `mapShaderNodeType` unchanged (identity fallthrough), and WGSLTranslator has no handler for that string. Every glass, rim-light, or view-dependent opacity material using the library's `utility_fresnel` node path still silently produces no output.

**Action:** Add `utility_fresnel: 'fresnel'` to `mapShaderNodeType`, or add a dedicated case in WGSLTranslator. Then add a test exercising the `MaterialLibrary` glass preset end-to-end through the translator.

## P3 — Style

### T-01 · `33bfffb0` · Commit cites `parseHolo` validation, not `validate_holoscript` MCP tool

**File:** commit message for `33bfffb0`

The message reads: "Both scenes parse cleanly via parseHolo." CLAUDE.md affirmation protocol requires `validate_holoscript` (the MCP tool) to be called on all generated HoloScript code, not raw `parseHolo`. The distinction matters because `validate_holoscript` runs the full validation pipeline including trait schema checking, whereas `parseHolo` only confirms syntactic parse success. The scenes are example files and are very likely correct, but the affirmation gap leaves trait-level schema errors undetected.

**Action (for future stress-test commits):** After `parseHolo`, also call `validate_holoscript({ code: <source> })` via the MCP server and include the result in the commit description.
