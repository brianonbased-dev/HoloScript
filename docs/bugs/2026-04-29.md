# Bug Report — 2026-04-29

## BUG-001 · Dockerfile `--no-dts || true` silences JS build failures

**File:** `packages/studio/Dockerfile` (lines 79, 85, 87–99)  
**Severity:** P2 — Silent production deployment failure  
**Status:** Open on `main`; fixed in `bug-hunt/2026-04-29` branch (related F-008 variant)

### What is wrong

Fourteen package build steps use `npx tsup --no-dts || true`. Since `--no-dts` invokes esbuild (not the TypeScript type-checker), the only failures it can produce are:

- A TypeScript/JavaScript **syntax error** in the package source
- A **missing module** that esbuild cannot resolve

Both are legitimate build failures that should stop the Docker build. `|| true` swallows them, producing a container with an empty or missing `dist/` directory. The studio then deploys and fails at runtime with `Cannot find module` 500 errors — the same class of error these build steps were introduced to prevent.

**Affected packages** (all have `npx tsup --no-dts || true`):

```
packages/crdt-spatial
packages/config
packages/platform
packages/mesh          ← foundational studio networking
packages/llm-provider  ← critical: Brittney route dep
packages/r3f-renderer
packages/studio-plugin-sdk
packages/ui
packages/connector-core
packages/connector-appstore
packages/connector-github
packages/connector-railway
packages/connector-upstash
packages/connector-vscode
```

### Root cause

The `|| true` was inherited from an earlier Dockerfile pattern where `tsc --emitDeclarationOnly` was used (and legitimately failed on cross-package type gaps). When the Dockerfile switched to `npx tsup --no-dts` (esbuild-based, no type checking), the `|| true` was kept but is now incorrect.

### Recommended fix

Remove `|| true` from all `--no-dts` builds. If any package then fails in Docker, that failure is legitimate and should be resolved by:

1. Adding the missing dependency's build step before the failing package.
2. Fixing the import (if the module is unavailable in the Docker context).

Do NOT silence failures with `|| true` after `--no-dts`; fix the root cause instead.

```dockerfile
# Before (incorrect — silences JS build failures):
RUN cd packages/mesh && npx tsup --no-dts || true

# After (correct — fails loudly on JS errors):
RUN cd packages/mesh && npx tsup --no-dts
```

### Context

A related but more severe variant (F-008, where `|| true` was used WITHOUT `--no-dts`, meaning DTS generation was also silenced) existed in commits on the `bug-hunt/2026-04-29` branch and was fixed in commit `0a723ce` using the `--dts-only || true` split pattern.
