# Bug Report ŌĆö 2026-05-07

## B-001: EIP-712 hash silently truncates sub-1 prices to 0

**Severity**: High (security / financial correctness)
**Introduced**: commit `98b94e0` (feat(mesh): xsp6 Phase 2 chain-anchor scaffolding)
**File**: `packages/mcp-server/src/holomesh/signing/chain-anchor.ts:135`

### Description

`hashSettlementReceipt` converts `NegotiationQuote.price` to a `uint256` via:

```typescript
price: BigInt(Math.trunc(hashable.finalQuote.price)),
```

`Math.trunc(0.05)` is `0`, so any price less than 1 (e.g., the demo's `price: 0.05 USDC`)
is committed on-chain as `price = 0`. The same truncation applies to `slaSeconds`, though
in practice SLA values are always integers.

The mismatch means the on-chain EIP-712 anchor cannot be used to prove the negotiated price ŌĆö
a verifier who recomputes the hash from the receipt would get a different hash than the
on-chain tx calldata if they scale the price correctly.

### Why tests don't catch it

The only test that exercises a fractional price is the agentic-internet demo
(`agentic-internet-demo.test.ts`), which uses `price: 0.05`. It passes because
`MockSigner` is initialised with a `fixedTxHash` and ignores the actual EIP-712 hash
passed as calldata ŌĆö so the hash produced by `anchorSettlement` is never compared
to any expected value.

The signer unit tests in `signer.test.ts` use `price: 100` (integer), so truncation
is a no-op there.

### Root cause

`uint256` cannot represent non-integer values. The code needs a canonical integer
representation for prices, e.g.:

- **Micro-USDC** (├Ś10ŌüČ): `0.05 USDC ŌåÆ 50_000n` ŌĆö standard USDC on-chain denomination
- **Whole-unit only**: require `price` to always be an integer in `NegotiationQuote`
- **String field in EIP-712**: change `price` type from `uint256` to `string`

Any of these changes the hash output for existing prices, so existing anchored txs
(including the founder Trezor swap, commit `1ed8b10b4`) would need to be noted as
pre-fix.

### Minimal safe action before a design decision

Add a guard in `hashSettlementReceipt` that throws if the price is not a non-negative
integer, so callers get a clear error instead of a silently wrong hash:

```typescript
if (!Number.isInteger(hashable.finalQuote.price) || hashable.finalQuote.price < 0) {
  throw new Error(
    `hashSettlementReceipt: finalQuote.price must be a non-negative integer ` +
    `(got ${hashable.finalQuote.price}). Scale to the appropriate on-chain ` +
    `denomination (e.g. micro-USDC ├Ś10ŌüČ) before calling.`
  );
}
```

This does NOT change the hash for integer prices. The agentic-internet demo would need
to update its quote to `price: 50_000` (micro-USDC) alongside this guard.

### Affected code paths

- `hashSettlementReceipt` in `chain-anchor.ts`
- `anchorSettlement` (calls `hashSettlementReceipt`)
- `settleNegotiationWithAnchor` in `agent-negotiation.ts` (calls `anchorSettlement`)
- Any production use of chain-anchored settlement with fractional-USDC pricing
