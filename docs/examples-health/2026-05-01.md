## Failure Clusters (today's scan)

### 1. Array literal — missing commas between object elements
**Affected:** `nft-marketplace-basic.holo`, `nft-marketplace-advanced.holo`,
`agents/multi-agent-pipeline.holo`, `agents/observable-agent-pipeline.holo`  
**Error:** `Expected RBRACKET, got LBRACE` / `Expected identifier, got COMMA`  
**Pattern:** `chains: [{network:"base"}{network:"polygon"}]` — HoloScript allows comma-free array-of-objects syntax that the composition parser does not yet accept.

### 2. Inline object without body block
**Affected:** `site.holo`  
**Error:** `Expected LBRACE, got OBJECT`  
**Pattern:** `object "Navbar" @panel(tag: "nav") @tailwind("...")` (no closing `{ }` block) — parser requires a body after trait decorators.

### 3. `EQUALS` / `STRING` in property position
**Affected:** `product-viewer.holo`, `world-builder.holo`  
**Error:** `Expected COLON, got STRING` / `Expected OBJECT, got IDENTIFIER`  
**Pattern:** Property assignment using `=` instead of `:`, or object declarations without keyword — same as yesterday's cluster 2.

### 4. Generic type syntax in implicit composition
**Affected:** `agents/distributed-npc-state.holo`  
**Error:** `Expected identifier, got LESS`  
**Pattern:** `Task<string>` TypeScript-style generics inside composition body.

## Recommended Next Actions

1. **Parser:** Accept comma-free array-of-object literals `[{...}{...}]` — affects NFT/multi-agent examples most.
2. **Parser:** Allow inline object declarations without body block (single-line `@trait` objects).
3. **Parser:** Reject `<` in identifier position with better error: "Did you mean to use a type annotation?" (not a parse crash).
4. **Parser → already known:** `=` property syntax, `->` reactive subscriptions.
